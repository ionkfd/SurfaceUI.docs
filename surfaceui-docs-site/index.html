Fabric client library for Minecraft Java 1.21.11

Build modern Minecraft HUDs and GUI screens with a real UI framework.

SurfaceUI combines a custom 2D renderer, layout engine, theme system, widget model, and input pipeline. It is inspired by the ergonomics of web UI, but it is not HTML, not browser CSS, and not a DOM.

Install

Use SurfaceUI as a Fabric library mod

SurfaceUI is distributed as a normal Fabric mod jar. Other Fabric mods depend on it at runtime and call the public facade from their client initializer.

  • Target: Minecraft Java 1.21.11
  • Runtime: Fabric Loader and Fabric API
  • Mappings: Yarn-named classes and methods
  • Language: Java 21
  • Side: client-only rendering and input code stays isolated
dependencies {
    modImplementation "dev.surfaceui:surfaceui:0.1.0"
}

// In fabric.mod.json
"depends": {
  "surfaceui": ">=0.1.0"
}

Architecture

Layered by responsibility

Renderer Layer

Owns drawing primitives: rounded rectangles, outlines, gradients, text, images, shadows, and clipping. Widgets call SurfaceCanvas, not raw vanilla rectangles.

Style Layer

Defines immutable style values and reusable theme tokens for colors, spacing, borders, radii, opacity, text, and fonts.

Layout Layer

Resolves anchor placement, fixed size, auto size, percent size, padding, gap, row, column, stack, min/max, z index, and clipping rectangles.

Widget Layer

Defines reusable components with ids, bounds, children, style, input traits, render hooks, and event handlers.

Input Layer

Routes mouse move, click, release, drag, focus, hover, and keyboard input to the correct widget, including drag capture.

Runtime Layer

Coordinates registries, config overrides, style resolution, layout, z sorting, input, rendering, and active mode.

Quickstart

Register your first HUD widget

Call the API from your mod's ClientModInitializer. Each provider returns a widget tree for the current frame, so text and progress values can be dynamic.

public final class ExampleClient implements ClientModInitializer {
    @Override
    public void onInitializeClient() {
        HudGui ui = HudGuiLibrary.api();

        ui.registerHud(Identifier.of("example", "clock"), ctx ->
            Widgets.row()
                .id("example:clock")
                .anchor(Anchor.TOP_LEFT)
                .offset(8, 8)
                .style(Style.create()
                    .padding(6)
                    .gap(5)
                    .radius(5)
                    .background(Paint.argb(0xCC101219))
                    .border(1, Paint.argb(0x66363B48))
                    .color(Paint.argb(0xFFE8ECF4))
                    .build())
                .add(Widgets.text("Day"))
                .add(Widgets.text(() -> Long.toString(day()))));
    }
}

Public Facade

The API other mods use

HudGuiLibrary

HudGuiLibrary.api() returns the singleton facade. HudGuiLibrary.runtime() exposes runtime services for advanced integrations.

HudGui

Registers HUD widgets, overlays, popup factories, screens, themes, and shared styles.

HudGuiApi

The concrete implementation delegates to HudGuiRuntime and keeps the public API small.

HudGui api = HudGuiLibrary.api();
api.registerTheme(myTheme);
api.registerStyle(Identifier.of("example", "panel"), panelStyle);
api.registerOverlay(Identifier.of("example", "warning"), 100, ctx -> warningOverlay());
api.registerPopup(Identifier.of("example", "toast"), ctx -> toastPopup());
api.openScreen(() -> new MySurfaceScreen());

Places where widgets render

Style System

Explicit, immutable, predictable

SurfaceUI styles are Java value objects. They are not browser CSS. There are no selectors, cascade rules, inherited surprises, or layout side effects. Themes provide defaults, and widgets override what they need.

Renderer

Custom 2D primitives over DrawContext

Widgets should render through SurfaceCanvas. Vanilla rectangle helpers are isolated inside the backend where needed. This keeps all widgets using the same primitives and allows the renderer to improve without rewriting widgets.

SurfaceCanvas canvas = new DrawContextSurfaceCanvas(context);
canvas.pushScissor(clip);
canvas.drawRoundedRect(rect, Radius.all(8), Paint.argb(0xEE111219), 1.0F);
canvas.drawRoundedOutline(rect, Radius.all(8), 1, Paint.argb(0x55363B48), 1.0F);
canvas.drawText("Ready", rect.x() + 8, rect.y() + 6, textStyle);
canvas.popScissor();
drawRectFlat filled rectangle for simple fills and rules.
drawRoundedRectRounded mesh or backend primitive with real radius values.
drawRoundedOutlineSubtle border around rounded shapes.
drawGradientRectGradient primitive for themes that use it.
drawTextFont renderer abstraction with measurement support.
pushScissorClip children so scroll and panels do not bleed.

Layout

Measured bounds first, rendering second

  1. MeasureEach widget computes a preferred size from content, style, and explicit size specs.
  2. PlaceRoot widgets use anchors and offsets. Flow containers place children in rows or columns.
  3. ClipEvery resolved widget carries a clip rectangle created from parent content and child bounds.
  4. SortZ index determines final render order.
Widgets.stack()
    .id("example:scroll_view")
    .size(260, 160)
    .style(Style.create().padding(0).build())
    .add(Widgets.column()
        .id("example:scroll_content")
        .offset(0, -scrollOffset)
        .size(248, contentHeight)
        .style(Style.create().gap(5).build()));

Input

Events route to the correct widget

containsHit testing uses widget bounds plus clipping.
hoveredUpdated on mouse move and stored by widget id.
pressedSet on mouse down, cleared on release.
focusFocusable widgets receive keyboard events.
drag captureSliders continue dragging after the cursor leaves the original bounds.
click contractButtons trigger on release inside after a valid press.

Config

User edits are small JSON overrides

SurfaceUI stores user-edited widget state under the Fabric config directory. Widget definitions remain code-owned; config only overrides anchor, x, y, width, height, visibility, and lock state.

{
  "overrides": {
    "example:mana_bar": {
      "widgetId": "example:mana_bar",
      "anchor": "TOP_RIGHT",
      "x": -12,
      "y": 12,
      "width": 120,
      "height": 34,
      "visible": true,
      "locked": false
    }
  }
}

HUD Editor

Interactive editing for HUD widgets

The editor is a client screen that renders the same HUD widgets in HUD_EDIT_MODE. Editable widgets can be selected, moved, resized, snapped to grid, inspected, reset, and saved.

Editor behavior Selection outlines and resize handles are rendered over widgets. Dragging writes a WidgetOverride. Releasing the mouse saves JSON. The reset button removes the override and returns to code defaults.

Fabric Integration

Client-only by design

Client entrypointInitializes the runtime and registers client hooks.
HUD renderingUses Fabric HUD rendering events to draw SurfaceUI HUD and overlay surfaces.
DrawContextThe low-level backend receives Minecraft's active DrawContext.
Screen subclassesFull GUI screens are regular Minecraft Screen implementations.
Yarn namesCode uses Yarn-named Minecraft classes and methods.
MixinsAvoided for core UI. Use only for optional integration points that Fabric events cannot expose.

API Reference

Core classes and responsibilities

HudGuiRuntime

Owns registries, theme state, config manager, render pipeline, input pipeline, and active mode.

SurfaceRegistry

Stores HUD, overlay, popup, and screen providers by identifier.

WidgetRegistry

Stores reusable widget factories and shared widget definitions.

RenderPipeline

Collects widgets, resolves styles, applies config, lays out, sorts, and renders.

InputPipeline

Tracks hover, focus, press, drag, resize, selection, and keyboard routing.

ZIndexSorter

Orders resolved widgets so overlays and popups draw predictably.

LayoutEngine

Converts widget trees into resolved rectangles and clips.

SurfaceRenderer

Draws resolved widgets through SurfaceCanvas.

ConfigManager

Loads, saves, updates, and resets user widget overrides.

Patterns

Recommended ways to build real interfaces

  • Keep gameplay/module state outside widgets. Widgets read and write state through suppliers and setters.
  • Use stable widget ids for anything the user can edit or persist.
  • Use rows and columns for most layout, stacks for overlays and scrollable content.
  • Let theme values define repeated color, radius, padding, gap, and text sizes.
  • Use CustomRenderWidget only for new primitives or special visuals, not for entire screens.
  • Use parent clipping for scroll areas, dropdowns, and panels that must not bleed.

Performance

Fast enough for every frame

Predictable passesCollect, style, config, layout, input, render.
Small objectsResolved widget records are lightweight and frame-scoped.
Text measurementText size is measured through the font abstraction and can be cached by the backend.
ClippingScissor rectangles prevent expensive overdraw in scroll areas.
No DOMThere is no browser-like tree diffing or selector matching.

Limits

Intentional boundaries

  • SurfaceUI does not implement browser CSS, selectors, HTML, or a DOM.
  • SurfaceUI does not use vanilla button widgets or vanilla rectangle rendering as the public widget renderer.
  • Mixins are not required for the core framework.
  • Advanced text shaping and full international typography are future renderer work.
  • Very complex screens should split state, widgets, and render helpers into separate classes.
