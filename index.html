Minecraft Java 1.21.11 - Fabric client library

Modern HUDs and GUI screens without hand-drawn layout math.

SurfaceUI lets Fabric mods register HUD widgets, overlays, popups, and full screens through Minecraft-native widget trees, immutable styles, and predictable layout passes.

Install

Add SurfaceUI as a client-side library

SurfaceUI is designed for Fabric mods to depend on as a normal mod/library jar. It targets Java 21, Minecraft Java 1.21.11, Yarn mappings, Fabric Loader, and Fabric API.

  • Client entrypoint: dev.surfaceui.client.SurfaceUiClient
  • HUD hook: Fabric HudElementRegistry
  • Renderer: Minecraft DrawContext
  • Screens: normal Minecraft Screen subclasses
repositories {
    maven { url = "https://maven.fabricmc.net/" }
}

dependencies {
    modImplementation "dev.surfaceui:surfaceui:0.1.0"
}

Quickstart

Register a HUD widget

Use HudGuiLibrary.api() from your client initializer. Widget providers run when SurfaceUI collects surfaces for a frame, so dynamic values can be supplied with lambdas.

public final class MyClient implements ClientModInitializer {
    @Override
    public void onInitializeClient() {
        HudGui hudGui = HudGuiLibrary.api();

        hudGui.registerHud(Identifier.of("demo", "clock"), ctx -> Widgets.text(() -> "Day " + day())
            .id("demo:clock")
            .anchor(Anchor.TOP_LEFT)
            .offset(8, 8)
            .style(Style.create()
                .background(Paint.argb(0xAA101820))
                .padding(5)
                .radius(4)
                .color(Paint.argb(0xFFE8F2FF))
                .build()));
    }
}

Core Concepts

Surfaces, widgets, styles, state

Surfaces

HUD, OVERLAY, POPUP, and SCREEN describe where a widget tree is hosted and how input should be routed.

Widgets

Widgets are Minecraft-native Java objects. They expose style, layout, bounds, input traits, optional children, and render behavior.

Modes

PLAY_MODE renders HUDs, HUD_EDIT_MODE enables selection and editing, and SCREEN_MODE captures input through screens.

Runtime State

Bounds, hover, focus, selected widget, drag state, resize state, and user overrides live in runtime/config layers rather than widget definitions.

Layout

Predictable layout without browser CSS

Fixed size.size(120, 34) or Style.width(120)
Auto sizeText and containers measure their content plus padding.
Percent sizePercent values resolve against the parent rectangle.
Row/columnChildren flow with explicit gap and margins.
StackChildren share a content rectangle and are layered by z index.
AnchorsRoot widgets can anchor to corners, edges, or center.
Widgets.row()
    .id("demo:status")
    .anchor(Anchor.TOP_RIGHT)
    .offset(-12, 12)
    .style(Style.create().padding(6).gap(4).build())
    .add(Widgets.icon(STATUS_ICON).size(12, 12))
    .add(Widgets.text(() -> "Ready"));

Styles

Reusable immutable design tokens

SurfaceUI styles are explicit value objects. There are no selectors and no browser cascade. A widget gets the active theme style for its type, then its own style is merged on top.

panel accent text stroke 
Theme compact = new Theme("my_mod:compact")
    .put("panel", Style.create()
        .background(Paint.argb(0xCC101820))
        .border(1, Paint.argb(0x553A5068))
        .radius(5)
        .padding(6)
        .build())
    .put("text", Style.create()
        .fontSize(9)
        .color(Paint.argb(0xFFE8F2FF))
        .build());

HudGuiLibrary.api().registerTheme(compact);

Runtime

Frame lifecycle

  1. CollectSurface providers create root widgets for HUD, overlays, and popups.
  2. Resolve StyleThe active theme and widget style merge into a single style object.
  3. Apply ConfigUser-edited positions, sizes, visibility, and lock state override defaults.
  4. LayoutAnchors, auto sizing, percent sizing, min/max, row, column, and stack layout resolve bounds.
  5. InputMouse and keyboard events route to hovered, focused, selected, dragged, or resized widgets.
  6. RenderResolved widgets render in z order through DrawContext.

HUD Editor

Built-in editing mode

The editor is a normal Minecraft Screen. It does not pause the game, draws a grid over the HUD, and saves config changes when the user releases the mouse or closes the screen.

Controls Click editable widgets to select. Drag to move. Drag the bottom-right handle to resize. Press R or the inspector reset button to restore defaults.

Config

User overrides are saved as JSON

SurfaceUI stores only user edits, keyed by widget id. Code defaults remain the source of truth, and config applies as a small override layer during layout.

{
  "overrides": {
    "example:resource_bar": {
      "widgetId": "example:resource_bar",
      "anchor": "TOP_LEFT",
      "x": 428,
      "y": 16,
      "width": 120,
      "height": 34,
      "visible": true,
      "locked": false
    }
  }
}

API Reference

Main entry points

HudGuiLibrary

api() returns the developer facade. runtime() exposes the active runtime for advanced integrations.

HudGui

Register HUD widgets, overlays, popup factories, themes, reusable styles, and open SurfaceUI-backed screens.

Widgets

Factories for text, box, image, icon, progress bar, row, column, stack, spacer, info HUD, and custom render widgets.

HudGuiRuntime

Owns registries, active theme, config manager, render pipeline, input pipeline, mode, and last render plan.

Common patterns

HudGuiLibrary.api().openScreen(root);

Limits

What SurfaceUI intentionally avoids

  • No browser DOM, selectors, cascade, or web technologies.
  • No mixins for core rendering. Mixins are reserved for optional deep vanilla integration.
  • No full text shaping engine. Custom fonts are Minecraft resource identifiers.
  • No expensive retained scene graph. Render plans are lightweight and rebuilt predictably.
